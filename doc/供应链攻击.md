## 研究背景与意义

### 硬件供应链攻击

### 软件供应链攻击

软件供应链 (Software Supply Chain, SSC) 指是指软件的整个生命周期，从选择和获取第三方组件、构建和集成软件、通过构建工具生成可执行文件，到通过分发平台或渠道发布和更新软件的过程。工业、政界、学术界计算机系统皆依赖于计算系统依赖于开放源代码和闭源软件组件的供应链。
软件供应链攻击在软件的源码、构建、分发等各个环节注入恶意代码，以便进一步危害链条下游的依赖系统。供应链攻击在现实中的高调发生远早于它的明确定义共识。[5,149] 将供应链攻击描述为将恶意代码注入供应链，以针对下游环节。这被认为是不完备的，如 [127] 识别到了对未维护的遗留脆弱代码的攻击。C. Okafor 等人 [156] 将供应链攻击描述为四阶段攻击模式：发现并利用供应链弱点、更改软件供应链、传播攻击者引入变化、利用下游链路的更改。该定义否定了对供应链单个软件的攻击引发的下游服务影响为供应链攻击。
本文将现有安全媒体所认为典型的供应链攻击皆认定为符合供应链攻击定义，并将这些供应链攻击类型分类，同样符合其攻击模式的攻击亦可认为是供应链攻击。

### 基于供应链攻击的现实案例讨论典型供应链攻击的攻击环节

基于 [134,135] 的供应链数据集与专业媒体近年的供应链事件报道，选取其中被报道为典型的供应链事件案例进行分类列举。

非开源软件的供应链攻击相当地复杂与多元化，攻击环节较为分散。

#### 代码签名证书环节的供应链攻击

2012年，Adobe 证书服务器被入侵，合法的 Adobe 证书被用于签署恶意软件文件，尤其是其中的 pwdum7v71 可以从 Windows 操作系统中提取密码哈希值[16]。
2012年，安全公司 Bit9 的代码签名密钥被窃取，被盗证书被发现签署了至少32个恶意软件包，1000多家公司和美国政府系统作为 Big9 的用户受到波及[17]，这起事件令业内反思白名单机制的隐患。
2014年，美国第二大零售商 Target 的网络凭证经由一家制冷分包商被窃取，攻击者将恶意软件推送至Target 的大多数 POS 设备上，窃取了约4000万个支付账户的信息，对 Target 造成高达4.2亿美元的损失 [18]。
2023年4月，Money Message 勒索软件团伙声称侵入了 PC 组件制造商微星公司并窃取大量数据，包括固件签名和启动保护密钥，意味着攻击者理论上可以创建并分发被恶意修改的 UEFI 固件，而这些固件会被受害者的计算机识别为合法更新 [121]。
这些案例是典型的代码签名证书环节的供应链攻击，影响了软件供应链中最基本的信任机制。

#### 设备制造和出厂前的软件安装时进行的供应链攻击

2015年，作为预装软件的一部分安装在联想笔记本电脑上的广告软件 Superfish 替换了用户的浏览器证书，通过篡改用户的 HTTPS 流量来插入广告，破坏了浏览器与网站之间的安全连接，并引入了潜在的中间人攻击漏洞 [19]。这是典型的在设备制造和出厂前的软件安装时进行的供应链攻击。

#### 第三方脚本或组件的提供环节进行的供应链攻击

2018年7月，来自 Magecart 组织的攻击者至少一次通过 JavaScript 漏洞篡改了 TicketMaster 网站上来自 Inbenta、PushAssist、Clarity Connect 和 Cloud Annex 四家公司的第三方代码，以盗取客户的支付信息，至少有800个其他电子商务网站的代码也遭到了类似的篡改，40,000名 TicketMaster 客户受到了影响 [37]。
2019年3月，Google Play 的206个总下载量达到近1.5亿次的安卓应用程序被发现感染了恶意软件 SimBad，该恶意软件通过驻留在名为 RXDrioder 的广告 SDK 中传播，可以在应用程序外显示广告，打开应用程序商店并引导用户下载其他应用程序，隐藏其图标以阻止卸载，打开 URL，甚至下载其他恶意软件和应用程序 [53]。
同为2019年3月，被称为“绵羊行动”的大规模数据窃取活动被披露，攻击者通过在数据抓取 SDK SWAnalytics 中插入恶意代码，入侵了12个在中国主要第三方应用商店发布的安卓应用程序，实现了超过1.11亿次应用程序下载，每当检测到手机重启或打开受感染的应用程序时，代码就会将用户联系人列表、登录信息、系统规格和地理位置上传到杭州顺网科技的 C&C 服务器 [54]。
2019年7月起，中国全球移动广告平台 Mintegral 的感染了 SourMint 的恶意 SDK 开始潜伏在苹果商店中，使用此 SDK 开发 iOS 应用程序时，部署的应用程序会感染恶意代码，从而实施广告归因欺诈，到2020年8月中旬被曝光为止，该恶意 SDK 被用于1200多个 iOS 应用程序，其中约70个应用程序被列入 App Store 免费应用程序前500名 [57]。
2020年6月，网络协议公司 Treck Inc 开发的广泛使用的低级 TCP/IP 软件库中被发现一系列零日漏洞 Ripple20，包括多个远程代码执行漏洞，影响了约有30家不同的供应商的数亿台设备，涉及工业和医疗设备、电网、石油和天然气等多个行业 [61]。
2023年5月至6月，Google Play 商店先后有 101 与 193 个安卓应用程序（包括多个视频编辑器和文件传输应用程序）被发现感染了一个名为 SpinOk 的广告 SDK 中的恶意软件，该恶意软件包含在一个提供每日奖励的小游戏的 SDK 中，能够搜索文件、渗入文件和修改剪贴板内容，这些内容可用于提取用户凭据或支付信息，被感染的应用程序已被下载合计超过4亿5千万次 [122,123]。
这些案例是典型的在软件的第三方脚本或组件的提供环节进行的供应链攻击。

#### 编译构建或部署阶段的供应链攻击

2015年，"XcodeGhost"通过非官方的恶意Xcode编译工具传播到中国各大App [2]，使用被入侵的 Xcode 制作的应用程序通过了苹果商店的安全检查，微信、12306、高德地图等均遭到污染，受影响的用户超过5亿。
2020年8月，有 Xcode 项目中被发现感染了恶意软件 XCSSET [62]，该恶意软件篡改了Xcode 项目文件（如.pbxproj），攻击者在项目构建中插入恶意行，一旦系统构建受污染的项目就能利用 Data Vaults 与 Safari WebKit 的零日漏洞窃取用户数据，还能窃取包括 Evernote 内容、Notes 信息以及 Skype、Telegram、QQ 和微信应用程序中的通信内容。
2022年4月，攻击者获得了 TravisCI 和 Salesforce 子公司 Heroku 的 OAuth 令牌访问权限，从而可以验证对 GitHub API 的调用，利用这种访问权限，攻击者有选择性地攻击 Heroku 和 Tracis-CI 应用程序的客户和用户的私有仓库和加盐散列密码 [95]。
这些案例是典型的编译构建或部署阶段的供应链攻击。

#### 软件分发环节供应链攻击

2016年3月，BitTorrent 客户端 Transmission 官网遭到攻击，其2.90版本的安装包被篡改为重新编译的，由一家土耳其公司签名的版本，被植入针对 macOS 用户的勒索软件 KeRanger [20,162]。
2016年7月，安卓第三方商店的 Pokemon GO 被嵌入可以访问通信和地理位置数据的恶意软件 DroidJack [24]。
2017年5月，Check Point Research 发现了一个通过字幕文件进行的攻击活动，攻击者上传的字幕文件通常由第三方网站托管，与盗版电影一起使用，其中包含恶意软件，普通媒体播放器可以自动下载这些文件，使恶意软件能够完全控制设备，而所有最流行的媒体播放器都存在漏洞，在更新发布之前，已有超过 2 亿用户受到攻击 [27]。
2017年7月，勒索软件 NotPetya 通过乌克兰的会计软件 MeDoc 更新传播，包括马士基和默克公司等数十家大公司遭到感染，造成的损失超过 100 亿美元，被认为是最具破坏力的网络攻击之一 [1,163]。
2017年10月，Eltima Software 的 Mac OSx 版流行媒体播放器 Elmedia 的官方服务器被入侵，发布了含有恶意软件 Proton 的篡改安装包 [64]。
2018年2月，BitTorrent 客户端 MediaGate 的一个更新服务器被入侵，攻击者在其安装程序中植入后门 Dofoil，将加密矿机下载到受影响的系统上，影响了约50万用户 [31]。
2018年7月，韩国遭到“红色签名行动”攻击，攻击者利用其先前窃取的远程支持解决方案提供商提供的代码签名证书，将签署的恶意软件植入更新服务器，针对韩国组织的服务器与数据库安装多种恶意软件 [38]。
2018年8月，拥有超过22万安装量的 Firefox 流行插件 Web Security 被发现秘密地将用户的浏览历史记录发往德国服务器 [41]，Mozilla 在后续发现并删除了23个含恶意行为的插件，这些插件使用了复杂的混淆技术来隐藏其恶意意图 [42]。
2018年8月，AppleJeus 行动被报告 [45]，与 APT Lazarus 有关联的攻击者通过看似真诚的加密货币交易程序 Celas Trade Pro 在更新中推送恶意程序 FallChill [44] 后门。
2019年1月起，攻击者发布了500个 Google Chrome 浏览器扩展程序秘密将隐私浏览数据上传到攻击者控制的服务器，它们至被全部删除为止已被下载了数百万次 [51]。
自2016年至2019年，与 APT 32（即海莲花 OceanLotus）有关联的攻击者在 Google Play Store 上开始了长期活动 PhantomLance，使用各种技术将间谍软件伪装为浏览器清理程序，这些应用程序最早在2018年才被检测到样本 [43]。
2020年11月，Lazarus 利用韩国帮助管理访问政府或互联网银行网站时所需额外安全软件的集成安装程序 WIZVERA VeraPort 展开进攻，攻击者攻陷使用额外安全软件的合法网站并替换其交付给 WIZVERA VeraPort 用户的软件，并利用了 WIZVERA 默认配置下验证签名时不验证证书归属的特性使用任意窃取的证书进行签名[65]。
2021年4月，ClickStudios 的 Passwordstate 密码管理器的就地更新机制被劫持，将恶意 dll 注入到分发给用户的更新包中，影响了多达29,000家受害公司 [74]。
2021年7月，勒索软件组织 REvil 利用 Kaseya VSA 中的多个零日漏洞大规模攻击，利用更新推送机制植入恶意代码，导致大量托管服务提供商及其下游客户的系统被感染 [76]。
2021年9月，WordPress 附加组件的开发商 AccessPress 被发现其所有免费插件和主题均被植入 PHP 后门，共影响40个主题和53个插件 [83]。
2022年5月，Saicoo 的用于读取通用访问卡（CAC）的 USB 设备（推荐给美国政府雇员使用的众多 USB 设备之一）被发现需要使用含有 Ramnit 恶意软件的驱动软件 [101]。
2022年9月，流行的 Magento-Wordpress 扩展供应商 Fishpig 遭黑客攻击，攻击者对 FishPig 开发环境的访问权限注入 PHP 代码下载 Rekoobe 远程访问木马，控制了 Fishpig 更新服务器，在 Fishpig 的所有付费扩展版本软件中注入了恶意软件 Rekoobe，软件的下载量超过 200,000 次 [108]。
2022年11月，有攻击者利用一个未公开的第三方广告网络或内容管理系统，将恶意代码注入到数百个美国新闻网站中，包括《洛杉矶时报》、《波士顿环球报》等在内的数百家美国新闻网站受到影响，当用户访问这些新闻网站时，恶意脚本会在用户的浏览器中运行。脚本会检查用户的设备类型和地理位置，然后选择性地下载 SocGholish 恶意软件 [112]。
2023年4月，针对中国用户的组织 Evasive Panda 被报告通过腾讯QQ消息应用程序自动更新传播恶意软件 MsgBot，记录键盘输入、窃取文件、捕获音频流等敏感信息，相关恶意活动最早可以追溯到2020年，并且主要影响了位于中国甘肃、广东和江苏省的用户，尤其是国际非政府组织的成员 [120]。
这些案例是典型的软件分发环节供应链攻击。
2020年3月，基础网络管理软件供应商 SolarWinds Orion 软件更新包通过被攻破的编译服务器被植入后门 SUNBURST，攻陷了多个美国联邦机构及财富 500 强企业网络 [3]，本次攻击针对软件供应链的构建、分发等多个环节，被认为是史上影响最广、最复杂的黑客攻击事件。

#### 开源软件的供应链攻击

随着软件项目对开源或专有模块的依赖急剧增加，SSC 变得越来越重要。
相较于非开源软件，开源软件的供应链攻击环节显得更加集中与统一。大部分攻击会通过第三方组件或库，或通过软件发布过程展开。CI/CD 系统也成为一个重要的攻击面，尤其是开源项目在这些平台上进行自动化构建时。

2016年，npm 生态系统中的一个关键依赖包 left-pad 被作者 Azer 删除，导致大量项目无法正常工作，包括 Kik Messenger 等知名应用 [21]，这揭露了开源模块供应链的脆弱性。在此背景下，针对开源代码的SSC攻击也因而被关注。
2016年，Linux Mint 官方网站的ISO链接及其校验和被篡改，植入 OpenSSH backdoor 后门的操作系统被攻击者用于构建僵尸网络 [15]。
2017年5月，开源视频转码软件 HandBrake 的 Mac 版软件镜像下载服务器遭到入侵，导致软件被植入后门 Keydnap，发布网站与 Homebrew 版本皆被篡改 [25]，Panic 公司的联合创始人 Steven Frank 因安装和运行了后门版本的 HandBrake 被攻击者窃取源码库并遭到勒索 [26]。
2018年4月起，Webmin 的开发服务器被入侵，并在6月发布的1.890版本中植入提权后门 [39]，到该后门于2019年9月发现为止，默认开启后门的该 Webmin 版本已被下载超过421,000次。
2018年5月，npm 下用户 dustin87 编写的植入后门的恶意库 getcookies 被报告，该库的最终渗透目标 mailparser 在当时每周的下载量约为64,000次 [33]。
自2018年5月至9月，3个包含 webshell 的 npm 恶意包 plutov-slack-client、nodetest199 与 nodetest1010 被上传，它们被保留在 npm 平台超过一年，直至 2020 年 10 月才被删除，期间每个包皆被下载超过100次 [34]。
2018年10月，开源控制面板 VestaCP 被发现其部分版本的脚本将管理员用户的密码发送至 vestacp.com，进而向受害者传输恶意软件 ChachaDDoS 以用于 DDoS 攻击 [46]。
2018年12月，流行的 PHP 软件包管理器 PEAR 发现其官网一直托管着一个带有后门的恶意可执行文件 [50]。

##### 基于依赖混淆的开源软件供应链攻击

2021年2月，安全研究员 Alex Birsan 发现了一种新型攻击依赖混淆 (Dependency Confusion)，他发现了这些公司内部托管的依赖项名称，而这些名称在 npm、PyPI 或 RubyGems 等外部公共存储库中并未使用。当 Birsan 将与内部托管包同名的恶意测试包上传到开源软件源时，安装程序命令（例如 PyPI 的'pip'）通常会安装他的版本号更高恶意包，而不是公司内部的包。利用这一开源依赖性问题他成功入侵了35家公司，包括微软、Netflix、Yelp 和苹果 [71]。这一时掀起了依赖混淆攻击的热潮。
2021年3月，用户 remindsupplychainrisks 出于引起人们对软件供应链攻击风险的关注的目的在 PyPI 和 npm 软件源中充斥了近5,000个依赖关系混乱的软件包 [72]，同时模仿犯也涌现利用依赖混淆攻击插入恶意代码 [73]。
2022年4月，一家名为 Code White 的德国渗透测试公司仿冒4个合法的德国私营公司在 npm 部署恶意软件包，与私人仓库的软件包名称相同，但版本更高，旨在执行依赖混淆攻击，部署后门并提取了受害者的机器信息 [98]。
2022年6月，有攻击者使用名为 @core-pas/cyb-core 和 notreallyapackgetrustme 的恶意 npm 软件包，试图针对 CyberARK 开发人员发起依赖混淆攻击，并窃取受害者的密码、亚马逊 EC2 凭据和系统配置信息 [103]。
2022年10月，npm 的 API 被发现允许攻击者执行时间攻击检测包管理器中是否存在私有包，这使得依赖混淆对于攻击者更有可行性，且 GitHub 回应由于架构限制无法防止时间攻击 [111]。
2022年12月，一名攻击者利用 PyTorch 的依赖关系混乱，在 PyPI 上创建了一个与 PyTorch 的一个依赖关系同名的软件包 torchtriton，该软件包自动优先于 PyTorch 官方仓库中的版本，其中包含的恶意二进制文件使用了反虚拟机技术来躲避检测，不以明文形式发布，并从用户那里提取了大量敏感数据，每周的下载量超过 2300 次 [113]。
2023年6月，仓库劫持 (RepoJacking) 被披露，在 GitHub 上，当项目所有者更改用户名或仓库名称时，会创建一个重定向以避免破坏依赖关系。但是，当其他用户使用组织的前名称时，这些依赖关系就会被覆盖。这就为黑客提供了可乘之机，黑客可以在这些前链接上创建新项目，并加入恶意代码 [125]。

##### 恶意维护者的开源软件供应链攻击

恶意者潜伏意图或是通过社会工程成为流行开源软件的维护者本人的案例也已被发现多起，此类攻击者拥有长期的权限以进行恶意的布局。
2018年11月，攻击者 Right9ctrl 获得了每周被下载多达200万次的流行 JavaScript 库 Event-Stream 的合法控制权，并插入了对 Flatmap-Stream 库的依赖，其中包含的恶意代码会渗透到钱包应用程序 Copay 中，一旦进入，它就会窃取私钥和钱包信息，并将其发送给攻击者，从而产生加密货币利润 [48]。
2022年1月，npm 上的一对开源库 faker.js 和 colors.js 被它们的开发者 Marak Squires 引入了恶意提交，劫持了这些软件包中使用的代码，在最终产品中输出奇怪的代码和阴谋论，导致成千上万的用户依赖的任何包含这些库的项目变得无用 [88]。
2022年3月，npm 软件包 node-ipc 的维护者 RIAEvangelist 故意在该软件包中引入一个抗议乌克兰战争的模块 peacenotwar 的依赖，恶意代码针对的是俄罗斯和白俄罗斯的 IP 地址，这一滥用行为导致 npm 生态系统出现连锁故障，流行的 Vue.js 前端 JavaScript 框架的用户与 Unity 游戏引擎用户皆受影响 [90]。
2024年3月，在大量 Linux 和其他基于 Unix 的系统上 被广泛使用的数据压缩组件 XZ Utils 被揭露存在后门，攻击者利用此后门能够在易受攻击的系统上以根权限执行恶意代码 [10]。在这个例子中，攻击者花费超过一年的时间获取项目维护者的信任，结合其他社会工程的手段，成为 XZ 项目的官方共同维护者，并在 Google oss-fuzz 中获得负责该项目的主要联系人身份。这一案例的隐蔽性之强、影响之广泛为开源代码的安全敲响了警钟。数千个小型开源项目构成了大型系统的支柱，实施更好的软件供应链安全性和提高警惕性迫在眉睫。

### 服务供应链攻击

存在这样一些攻击案例，它们被一些软件供应链攻击的研究收录为案例，但却与典型的软件供应链攻击有较大区别。原因在于，这些攻击案例的攻击对象并非在传统的硬件或软件供应链上，本身不是软件下游的直接依赖环节，但是对象提供的关键的基础设施服务却被其他客户或软件依赖。在这个更广泛的供应链框架下，应被视为对服务供应链的攻击，通过攻击一个关键的“服务环节”来影响其所有下游客户。这些攻击有时相较于传统供应链攻击更直接、集中在服务本身。

2020年12月，攻击者利用 Accellion FTA 的零日漏洞，向服务器的 Web 服务根目录中插入了 webshell，攻击者可以窃取 Accellion 客户上传到该服务的文件[67,68]。它被一些归类为供应链攻击，但实际上它更可能直接被归类为MITRE ATT&CK的T1190: Exploit Public-Facing Application，或是运行时安全风险。

2021年3月，黑客通过盗取 Verkada 系统的管理员账户，进入了该公司云端视频监控平台 [160]。Verkada 为全球数千家企业提供视频监控服务，包括医疗、教育、政府等机构。黑客不仅访问了客户的视频监控流，还泄露了企业的敏感信息和设施的监控数据。ENISA 将这起事件视作供应链攻击的典型案例 [161]。这起事件本身更像是一个针对 Verkada 内部系统和账户的直接攻击，通过盗取管理员账户而非通过软件或硬件的供应链进行攻击。这种攻击通常称为 “内部威胁” 或 “凭证填充攻击”，黑客通过破解或盗取了管理员的凭证来获取对系统的控制。但是处于监控服务的敏感性，对其本身的攻击也对服务对象造成了扩散性影响。此处的供应链攻击定义争议在于，Verkada 并不处于传统的软件或硬件供应链上，虽然它为很多下游客户提供视频监控服务，但它本身并不作为下游应用的“依赖环节”存在。它提供的是一个独立的、专门化的服务，而不是某种软件库、API 或基础设施，这使得它在供应链攻击的讨论中不被视为供应链中的“中间环节”。但是转换视角，从“服务供应链”的角度来看，Verkada 也可以被视为供应链的一部分。Verkada 是一个提供关键基础设施服务的供应商，并且它的服务被多个客户（包括医院、学校等）依赖。这样，在这个更广泛的供应链框架下，攻击 Verkada 也可以视为一种对服务供应链的攻击，因为它通过攻击一个关键的“服务环节”来影响其所有下游客户。因此，这个事件可以被看作是一个服务层面的供应链攻击，但它通常不被归类为典型的供应链攻击，因为它的攻击目标更直接、更集中在 Verkada 服务本身，而不是通过破坏某个中间环节来影响整个生态系统。

### 研究的目的与问题阐述

根据目前供应链攻击的案例，可以发现供应链攻击涉及的恶意代码与常规的恶意软件存在显著不同。
常规的恶意软件追求低成本大规模的感染，通常以非专业用户为主要目标，这些用户可能缺乏足够的安全意识和技术知识，容易成为攻击的目标。为应对不同的目标环境，同一家族的常规恶意软件存在多样大量的变种，虽然也会尝试隐藏自身，但由于需要快速传播，隐蔽性相对较低。由于这些特点，常规恶意软件虽然影响广泛，但是就单次攻击而言，影响较小。
供应链攻击则不同，通过篡改供应链中的某个环节（如源代码、编译器、构建过程等）来植入恶意代码，追求一次成功植入，影响大量用户，主要针对的对象是专业用户或企业级用户，这些用户通常具有较高的安全意识，但供应链攻击利用了他们对供应链的信任，使得攻击更为隐蔽和有效。成功植入一次即可达到目的的特点，使得供应链攻击的恶意代码通常经过精心设计，具有更高的针对性，变种较少。供应链攻击追求更高的隐蔽性与持久性，它们通常会被数字签名或伪装成合法组件，使得检测难度增加。这些特点，使得供应链攻击的覆盖范围可能较小，但由于能够影响整个供应链中的所有用户，导致单次感染造成的影响极为严重。

## 供应链攻击

### 供应链攻击的安全威胁研究现状

2019年，IQT Lab 建立了一个基于公共报告的软件供应链安全漏洞的数据集，更新至2022年9月，并提出供应链安全的三个主要障碍：缺乏数据收集和分析、现有的应用安全产品无法识别软件供应链攻击的独特特征、减少软件供应链攻击面需要采用现有技术和流程以提供验证源代码和二进制文件的来源和内容所需的信息，从而消除或减轻许多被攻陷的风险 [134]。
2020年，V. Craciun 等人针对使用 SGX 完整性检查程序进行对基于代码篡改的供应链攻击防御方法，提出了一种拦截 SGX 签名过程的攻击方案，并提出了在安全区内实现签名器的集中式缓解方法与利用区块链智能合约的分布式缓解方案 [137]。
2021年，大西洋理事会构建了2010年以来的供应链事件数据集，至今仍在更新，并提出了三点主要建议：改善基线、更好地保护开源、反制系统性威胁 [135]。

#### 开源软件的供应链攻击研究

2020年，M. Ohm 等人收集、分类和手动分析了一个针对开源软件供应链的实际攻击软件包的恶意代码的数据集 Backstabber’s Knife Collection，基于对该数据集的分析，数据外泄、下载第二阶段有效载荷、反向shell、引发拒绝服务、运行加密矿工等以上行为是供应链恶意软件包的主要目标 [5]。
2023年，P. Ladisa 等人以 [5] 提出的攻击树为起点，基于系统文件回顾 SLR 识别了107个开源软件供应链攻击向量，主要分类为从零开始开发和宣传独特的恶意软件包、创建合法包的名称混淆、破坏现有合法包，其中最后一类攻击被分类为注入合法包源码、在合法软件包构建期间注入、分发合法软件包的恶意版本 [149]。

##### 针对开源软件代码分发平台的安全威胁研究

2018年，L. Gong 提出轻量级动态插桩监控分析系统 NodeSec 来检测 Node.js 应用中的安全风险，通过内置的 Node.js 系统函数捕获 JavaScript 与操作系统之间的交互，采用迭代半自动手工分析日志的方法隔离一组可疑的内置系统函数，研究了 npm 平台下超过330,000个包，发现了360个之前未知的恶意或易受攻击的包 [130]。
2018年，G. Mezzetti 等人研究了 npm 平台自动化下更新是否会引起破坏性更改的回归性测试 [128]。
2019年，M. Zimmermann 等人分析了 npm 平台的安全威胁，指出开发者对代码统一分发的需求引入的锁定依赖项技术 package-lock.json 导致了包的最新可用版本与依赖项使用的版本之间的技术滞后、基于微包的代码重用生态导致了 npm 的过大依赖数量与攻击面的扩大、Javascript 语言的特权分离机制缺乏与 Node.js 平台的沙箱缺乏导致第三方包的权限过大、npm 平台的系统审查流程缺乏、npm 的包发布与 Github 平台的脱节导致的账户混淆问题等安全风险特性，提出了恶意包 (TM-mal)、未维护的遗留代码 (TM-leg)、包接管 (TM-pkg)、账户接管 (TM-acc)、合谋接管 (TM-coll)五种威胁模型，并基于包依赖关系图、维护者、漏洞三种指标定量分析这些威胁的可实现性 [127]。
2021年，K. Champion 等人提出基于开源代码的“产出不足”情况测量开源代码的风险的方法，并将方法用于 Debian 软件包的测量，识别了至少4,327个软件包的生产不足 [131]。F. Massacci 等人提出了技术杠杆的概念，即从第三方库导入的代码大小（包括直接依赖于传递依赖）与编程语言标准库大小以及库自身代码大小之间的比例，将所提出的指标应用于 8464 个来自 Java Maven 的 FOSS 库实例 [132]。
2022年，N. Zahan 等人针对 npm 平台研究了开源软件供应链攻击的薄弱环节，提出了过期的维护者域名、安装脚本、不活跃维护、维护者过多、贡献者过多、拥有过多维护包的维护者六个供应链薄弱信号，其中前三种信号被从业者支持而后三者被反对 [6]。
2022年，Y. Wang 等人基于 Java 项目第三方库的使用强度、使用过时程度、更新强度和更新延迟等指标分析第三方库与安全漏洞相关的使用风险和更新风险 [126]。
2023年，S. Neupane 等人针对软件包混淆进行深入研究，基于主题分析的方法将软件包混淆分为13类（前后缀增强、序列重排、分隔符修改、限定范围混淆、语法替换、同语义替换、异语义替换、同音相似性、移除前后缀、替代拼写、近形替换、1步 Damerau/Levenshtein 距离、熟悉术语滥用），并开发了一种无分隔符的标记算法实现对除熟悉术语滥用与1步 Damerau/Levenshtein 距离以外的其他分类的包混淆检测规则，标记了360,333对 npm 包，并通过在线调查的方式评估了在真实场景中这些样本对于人类的混淆程度，发现序列重排、语法替换和替代拼写的高度混淆性，以及分隔符修改和限定范围混淆的潜在混淆性，而除同语义替换、非语义替换和同音相似性这三个表现最差的类别外，其他分类均表现出一定混淆性，最后通过安全性分析得出被混淆规则标记的包被 Snyk 数据库标记为恶意的可能性是未被混淆规则标记包的3倍 [133]。
2024年，S. Amft 等人基于半结构化访谈的方法研究了开源软件贡献者的个人安全设置问题，发现尽管开源软件开发者通常倾向于实用的安全设置，但开源软件领域普遍存在的信任和不干涉文化阻碍了安全知识和最佳实践的共享，并提出使用基于平台的安全机制而非单独提示贡献者，利用访问控制层次结构和 CI/CD 解决方案来限制具有特权访问或发布权限的账户数量等建议 [152]。

##### 针对开源软件代码构建平台的安全威胁研究

2022年，I. Koishybayev 等人针对 Github CI 工作流的安全性展开研究 [143]。
2024年，Y. Gu 等人 针对持续集成 (CI) 服务中的缓存安全展开研究，基于主流 CI 平台的源码分析与黑箱测试解释了持续集成服务的缓存实现细节，识别出现有 CIP 的缓存共享与缓存继承机制的隔离脆弱性，提出不当写入、不当读取、不当令牌权限、不当缓存驱逐四种攻击向量，并开发分析工具对开源代码库进行大规模测量 [150]。

### 软件供应链攻击的防御方法的研究与实践

2018年，N. Vasilakis 针对 Javascript 语言提供了一个运行时隔离模块平台 BreakApp [154]。由于需要开发者的努力，实用性受到限制。
2018年，Google 安全团队提出了 in-toto 框架，关注软件开发和构建过程中每个步骤的安全性，确保软件从源代码到最终交付物的整个过程都能被追溯和验证，使用加密签名认证确保每个步骤由受信任的实体执行，生成不可篡改的认证链，核心在于防止外部第三方对供应链的干扰，但不会解决供应链环节本身不可靠的问题 [138]。
2020年，开放源代码安全基金会 OpenSSF 建立并提出了开源软件供应链安全分级标准 SLSA，将安全要求分为四个级别，每个级别要求逐步增加安全性，项目可以根据自身的需求和成熟度逐步实施这些安全措施，侧重于构建环境和过程的可验证性，通过这些级别提供了一个渐进式的安全框架 [140]。
2020年，Google、Linux Foundation、OpenSSF 等多个组织推出了 Sigstore 项目，为开源软件和软件供应链提供简便、安全的签名机制，帮助减少软件供应链中的恶意攻击风险，提升软件的可信性 [142]。
2021年3月，OpenSSF 推出了工具 OpenSSF Scorecard，通过评估多个安全方面（如代码质量、依赖管理、构建安全、漏洞响应等），生成一个安全评分，帮助开源项目维护者改进安全实践，帮助企业和开发者选择更加安全的开源项目，为开源软件供应链的风险管理提供一个有效的工具 [141]。
2022年，IETF 提出供应链安全架构草案 SCITT，目标是提供供应链中各个环节跨组织、跨领域的整体信任模型，通过 Trust token 实现多个供应链参与方之间的的透明性、可追溯性和可信度验证，适合对发布过程进行透明验证的场景，不关注各个环节的细节 [139]。
2024年，F. Fischer 等人针对 Github 自2017年以来的脆弱依赖项安全警报、脆弱依赖项自动化安全更新、代码扫描漏洞警报三项安全干预措施进行有效性分析，基于 Javascript 与 Python 采样代码库的时间序列数据进行反事实预测，发现安全警报对用户手动安全更新的数量增长影响高达149%，自动化安全更新导致的安全更新数量增长32%，而代码扫描对漏洞修复的影响较低，在一些漏洞案例里甚至是微不足道的 [151]。

### 针对软件供应链攻击的恶意代码检测

#### 传统恶意软件检测面对软件供应链攻击的挑战

目前杀毒软件样本库中的绝大多数样本是传统恶意软件，尤其是通过非供应链方式传播的恶意软件，而供应链攻击的恶意软件样本相对较少。传统恶意软件检测对于软件供应链攻击面临独特的挑战。

##### 供应链攻击的隐蔽性和复杂性

供应链攻击通常比传统的恶意软件传播更为隐蔽，攻击者往往通过篡改开源包、依赖库、软件更新机制等途径渗透到目标系统。这些恶意代码通常在目标系统或开发环境内执行后才暴露出来，而不是像传统恶意软件那样通过明显的行为（例如恶意附件、恶意链接等）直接感染用户设备。对于终端用户来说，他们并不会接触到这些更新包或依赖包的内部内容。
供应链攻击通常需要高度的技术水平和资源，攻击者可能专注于少数特定目标，如大型公司或开源项目。因此，这类恶意软件的传播不如传统恶意软件广泛，也不容易在普遍的杀毒软件样本库中找到。
传统的恶意软件（如木马、勒索软件、病毒、蠕虫等）通常会有非常明显的恶意行为模式，如文件加密、CPU 占用率过高、远程控制、恶意广告弹出等。这些行为在目标系统中很容易被反病毒软件或入侵检测系统（IDS） 发现。
相比之下，供应链攻击中的恶意代码（例如篡改过的依赖包或软件更新）往往更难以直接检测到，因为它们通常是在开发和构建过程中插入，或者作为隐形依赖出现在最终用户的环境中。这使得供应链攻击的恶意软件样本更难进入传统的杀毒软件样本库。

##### 供应链攻击的目标通常是开发者或企业

供应链攻击恶意软件的直接目标往往是开发者、企业环境，而不是普通的终端用户。许多供应链攻击的典型例子（如 SolarWinds、Codecov、npm 包篡改等）都发生在开发人员或企业内部，目标是通过开发工具链、CI/CD 环境、开源依赖包等渠道渗透。
因此，这些恶意软件样本的传播通常较为有限，并不在普通用户的设备上出现，也不会直接暴露在传统的反病毒软件检测范围内。

##### 供应链攻击的恶意软件需要特殊检测机制

供应链攻击中的恶意软件可能不会表现出明显的恶意行为，而是潜伏在开发过程或运行时环境中。自动化构建工具、CI/CD 管道、包管理系统等常规的反病毒软件无法有效地检测到这些恶意软件。
这类攻击的检测需要专门的安全工具和技术，如软件成分分析（SCA）、代码签名验证、构建环境监控等。这些方法不依赖于传统的样本库，而是通过对开发过程和依赖管理的监控来发现潜在的风险。

##### 反病毒软件尚未完全适应供应链攻击

大多数传统 杀毒软件和反恶意软件工具主要依赖签名匹配和行为检测来检测恶意软件。而供应链攻击往往利用漏洞、篡改合法代码或依赖包，这些方式不容易通过常规的签名库或行为分析捕捉到。
随着供应链安全的重视，新型安全工具（例如 in-toto、SLSA、Sigstore 等）开始出现并针对这类攻击提供防护，但这些工具和技术尚未被普遍集成到传统的反病毒软件和终端安全产品中。

#### 针对软件供应链攻击的恶意代码检测的研究方法文献综述

2023年，M. Ohm and C. Stuke 分析了20篇设计供应链恶意软件包检测的论文并进行了分类比较，指出针对供应链攻击的一般静态分析方法由于缺乏恶意软件包示例与缺乏测试而不够成功，而由于安全问题和计算要求，动态方法应谨慎对待，没有一种通用的解决方案，而是可以在不同环境中使用多种不同的方法来检测不同类型的恶意行为 [157]。

## 讨论

### C 语言生态下的开源供应链安全研究不足

#### C 语言的底层作用：许多关键工具和库都依赖 C 语言

##### Python 底层

Python 本身虽然是一种解释性语言，但它的核心解释器（CPython）是用 C 语言编写的。因此，Python 代码的执行依赖于底层 C 语言的库（例如 libpython）。如果 C 语言的供应链受到攻击，可能会影响到 Python 解释器的安全性，从而影响到数百万开发者和生产环境。

##### 其他流行的工具和库

xz、gzip、bzip2 等压缩工具，它们大多数都是用 C 语言编写的，广泛用于操作系统和开发工具中。此外，C 语言也被用于很多 系统级工具，比如 OpenSSL、libc、curl、nginx、git 等，它们都是开源项目中的核心部分，涉及到许多操作系统、服务器和开发工具链。

##### 操作系统的核心组件

操作系统本身（如 Linux、Windows、macOS）的大部分核心组件也都是用 C 或 C++ 编写的。许多驱动程序、内核模块、系统库等都依赖于 C 语言，而这些代码如果在供应链层面受到攻击，可能会导致整个操作系统的安全性受到威胁。

#### C 语言的供应链风险与攻击

##### 构建工具和依赖管理

C 语言开发中常用的构建工具（如 Makefile、CMake、Autotools 等）对项目的构建过程起着至关重要的作用。攻击者可以通过篡改构建工具、伪造源代码或通过修改二进制文件来插入恶意代码，进而影响到整个开发环境的安全。

##### 二进制传播

与解释性语言相比，C 语言的编译型特性使得恶意代码更容易通过二进制文件传播。如果攻击者在构建过程中成功注入恶意代码，最终生成的二进制文件可能被广泛传播，而很难通过源代码分析直接发现问题。例如，一些编译后的 C 语言库可能会被不同的项目或工具所依赖，如果供应链中某个环节受到攻击，所有依赖该库的项目都可能暴露在攻击者的控制之下。

##### 构建与发布过程的隐蔽性

由于 C 语言的构建过程通常涉及到复杂的工具链和多个依赖项，攻击者如果能够篡改其中某些环节，可能会导致难以被检测的供应链攻击。例如，修改构建脚本、伪造源代码、篡改编译器或链接器等，都可能成为攻击的路径。

#### 对 C 语言生态的忽视可能带来的后果

如果 C 语言的供应链安全问题被忽视，可能会导致广泛的系统级漏洞，影响到从操作系统到应用程序的整个软件栈。比如，针对 C 语言程序的 缓冲区溢出漏洞、整数溢出、内存泄漏 等漏洞，可能被攻击者利用来获得系统权限或执行恶意代码。
许多重要的开源项目和基础工具都依赖于 C 语言编写，攻击者如果成功篡改这些工具中的代码，可能会在 全球范围内 影响数百万用户。例如，OpenSSL 中的著名漏洞 Heartbleed 就是通过 C 语言代码中的一个安全漏洞，影响了大量使用该库的系统和应用。
C 语言的构建和发布过程通常比较复杂，攻击者可以通过篡改构建脚本、伪造编译选项等方式，在不修改源代码的情况下插入恶意代码。这种攻击方式往往较为隐蔽，不容易被传统的静态分析工具发现，从而增加了检测和防护的难度。

#### C 语言供应链攻击研究不足现状

##### 研究资源不足

目前，关于 C 语言供应链攻击的研究相对较少，主要集中在一些传统的漏洞扫描和静态分析工具上，缺乏针对 C 语言生态的深入、安全审计。与 Python 和 JavaScript 的研究相比，C 语言的研究通常更侧重于漏洞分析和修复，而不是从供应链安全的角度进行系统性研究。

##### 统一标准与工具的不足

C 语言的生态缺乏像 PyPI 和 npm 这样的集中管理平台，使得供应链攻击的检测和防护更为分散。虽然有一些静态分析工具（如 Clang Static Analyzer、Coverity 等）可以用于检测 C 语言的安全问题，但缺乏一套统一的机制来检测供应链攻击的传播路径和影响。

#### C 语言生态供应链研究挑战

##### C 语言生态的分散性

###### 缺乏统一的包管理平台

像 PyPI（Python）和 npm（JavaScript）这样的包管理平台非常集中和统一，所有流行的第三方库和工具几乎都可以从这些平台上找到并安装。而对于 C 语言，虽然有一些包管理工具（如 Conan、vcpkg、Homebrew），但它们并不像 PyPI 和 npm 那样统一，且它们更多地面向特定平台或特定类型的库。

###### 依赖关系的复杂性

C 语言程序的依赖管理通常比较复杂，开发者可以直接从 GitHub、SourceForge 或个人网站下载库并手动集成，导致 C 语言包的发布和管理没有一个集中化的标准。这也意味着，很多 C 语言项目和库可能没有经历类似 Python 或 JavaScript 项目中那样严格的审查和集中化管理。

###### 构建和发布过程的差异

C 语言的构建通常通过 Makefile、CMake 等工具进行配置，这些工具的配置方式和发布方式各不相同，缺少统一的标准。这使得跨平台的构建和发行变得更加复杂，也让分析和审查 C 语言代码的过程变得更加多样化和碎片化。

##### 研究数据的缺乏

###### 缺乏集中式的恶意包数据库

在 Python 或 JavaScript 生态中，恶意包往往可以通过包管理平台（如 PyPI、npm）进行集中分析和汇总。研究者可以依赖这些平台提供的数据，如恶意包的下载量、用户评价、历史版本记录等。C 语言缺少类似的集中平台，因此很难对恶意包进行全面的跟踪和收集。

###### 分散的攻击数据和案例

在 C 语言中，恶意包和供应链攻击可能发生在不同的社区、公司、工具或开源项目中。研究者需要从多个来源获取不同的攻击案例和数据，这些数据往往没有统一的格式，也不容易进行对比和总结。例如，C 语言的开源库和工具可能有自己独立的依赖管理机制，攻击者也可能通过修改构建脚本、伪造源代码、修改二进制文件等多种方式进行攻击，而这些方式的案例不容易被集中的平台记录和共享。

##### C 语言的攻击和分析难度

相对于解释型语言，C 语言的构建过程和二进制依赖的复杂性使得安全研究变得更加困难。
C 语言是 编译型语言，它通常依赖于静态编译过程生成二进制文件。攻击者可以通过篡改构建过程、伪造编译器选项、修改二进制文件等手段来实施恶意注入。这些攻击形式比解释型语言（如 Python 或 JavaScript）中的恶意包注入要复杂，因此对 C 语言供应链攻击的研究相较于纯代码分析需要更高的技术要求和更复杂的工具链。
C 语言的构建和分析通常需要依赖一套复杂的工具链，包括编译器、构建系统、调试器等。这些工具的差异可能会影响攻击的传播方式和分析结果，使得对 C 语言的供应链攻击进行系统性的分析变得更加困难。
许多传统的安全方法（如代码审查、漏洞扫描、静态分析等）对于 C 语言的供应链攻击并不适用。研究人员和安全专家更倾向于将精力投入到其他更容易分析和防护的语言（如 Python、JavaScript）上，而忽视了 C 语言的潜在风险。
